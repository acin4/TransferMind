2. Current architecture snapshot
   Frontend responsibility: UI, routing, and backend HTTP calls only. There are no frontend source reads from Supabase; the app uses VITE_API_BASE_URL and fetches /api/... endpoints from api.ts (line 1).
   Backend responsibility: Express API, request validation, and read-side translation from DB rows to frontend-safe shapes. The route/controller/service/repository split is present in backend/index.js (line 1), backend/api/routes.js (line 1), controllers/, services/, and repositories/.
   Ingestion responsibility: external API fetches plus Supabase upserts in backend/ingestion. This is still script-driven, not orchestrated through the bootstrap/cron/on-demand flows described in the README.
   Database role: raw-ish storage tables plus read-side views. The useful views already in play are current_tournament_seasons (line 1), current_season_teams (line 47), and standings_with_team_info (line 1).
   Id model right now: frontend-facing routes use internal DB ids for teams and players; the backend translates where needed. The drift is in storage: teams/players have both id and api_id, but standings.team_id, standings.tournament_id, standings.season_id, and team_stats.team_id are still API ids, while players.team_id is moving toward internal ids. That mixed semantic is the main architectural debt.

4. What is problematic or incomplete
   -The bad fetchSeasons.js change is still present. It determines current season from only the first 3 rows and only upserts rows.slice(0, 3) in fetchSeasons.js (line 37). That means historical season coverage is still broken.
   -fetchPlayers.js is currently broken at runtime. fetchSquadPlayerIds uses undefined variables team*api_id and teamApiId in fetchPlayers.js (line 149), so the script will fail before loading squads.
   -fetchAllPlayerStats.js still assumes players.team_id === standings.team_id in fetchAllPlayerStats.js (line 142), but those columns now represent different id domains in practice. This makes player-stat ingestion fragile or wrong.
   -Stats are not actually scoped to current seasons as the README says. fetchAllTeamStats.js reads all rows from standings in fetchAllTeamStats.js (line 119), and fetchStandings.js reads all seasons from seasons in fetchStandings.js (line 13), not is_current only.
   -Standings multi-stage support is only half done. Ingestion stores standing_group*_ and stage*tournament*_, but the API read path ignores them and rewrites position using array index after sorting only by points in standingsRepository.js (line 31) and standingsService.js (line 31). Regular season vs playoff/playout/qualifying is therefore not represented correctly.
   -Team stats reads are ambiguous. getLatestTeamStatsByApiId (line 44) does .eq("team_id", apiId).limit(1) with no season filter and no deterministic ordering, so /api/teams/:id/stats can return an arbitrary row.
   -Frontend pages expect richer player data than the backend returns. Players.tsx (line 42) expects player_stats, and Teams.tsx (line 291) expects position plus rating-like fields, but /api/players currently returns neither.
   -There is a field-name mismatch in the team profile UI. TeamProfile.tsx (line 83) reads stats.clearances, while the stored/team API shape uses clearences.
   -The README is ahead of the code. It describes transfer-history, refresh endpoints, bootstrap/cron scripts, and folder layout that do not exist in the repo today in README.md (line 42), README.md (line 105), and README.md (line 290).
5. Important inconsistencies / technical debt
   Id mapping: the backend response layer is trying to expose internal ids consistently, but the ingestion/storage layer still uses API ids in several columns named generically as team_id, season_id, and tournament_id. playerService.js (line 39) compensates on reads, but ingestion scripts do not consistently follow the same policy.
   Ingestion correctness: you should fix ingestion before adding features. Right now the season source-of-truth is incomplete, player ingestion is broken, and player-stats joins are semantically wrong. Any transfer-history work built on top of this will inherit bad mappings.
   Views vs direct queries: the current views are the right direction for standings and current-scope resolution. Keep current_tournament_seasons, current_season_teams, and a standings read view in SQL. Keep simple team/player list/detail queries direct for now. Move “current/latest team stats by internal team id” into a DB view because the current direct query is under-specified.
   API design consistency: route shape is good and backend-centric, but the read contract is drifting from the frontend in player/team detail pages. Also, there are no on-demand refresh endpoints despite the README describing them.
   View/schema staleness risk: 20260329193451_remote_schema.sql (line 295) does not include the later standings stage columns added in alter_table_standings.sql (line 1). The migration chain is valid, but the snapshot is stale as documentation.
   Backend-only access drift: the frontend code no longer reads Supabase directly, but the DB migrations still grant public read access on several tables/views in add_standings_view_and_public_read.sql (line 5). That is not aligned with the stricter backend-only direction.
   Dev/test workflow gaps: frontend builds, but npm run lint fails because eslint is not installed even though the script exists in frontend/package.json (line 6). backend has a broken test:api script pointing at a missing file in backend/package.json (line 7). The root package.json (line 1) is just {}, so there is no clear top-level workflow. I could not verify live dev-server startup in the sandbox because binding to local ports is blocked here.
